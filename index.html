Lookup CF — AC Solutions
Option 1 — Force RCF to Grant Access to Linked Objects
Right now a user can see a lookup value without having access to the record it came from. This option fixes that by making the Reference field automatically grant every task member read access to the linked records — so anyone who sees the value is guaranteed to have legitimate access to its source. This is already how A4D works.

Pros

  • Clear UX change
  • No need to worry about bypassing AC because we'd be guaranteed anyone looking at the lookup field value would have access to the linked object's source field

Cons

  • Breaking change for existing RCFs, since it silently expands the audience

Option 2 — Per-field Proxy Actor Model
Instead of fetching the lookup value as you, the system fetches it as a designated proxy — usually the person who created the field. Everyone on the task sees the same value regardless of their own permissions, because it was retrieved using the proxy's credentials, not theirs. Asana Rules already work this way.

Pros

  • There is precedence to this (e.g., Rules)

Cons

  • Users have no mental model for "this field value was fetched on behalf of a proxy, not you." Need to surface this to them somehow.

Option 3 — Server-side AC Chaining (LCF → RCF)
When someone requests a lookup value, the server walks the chain — Lookup field → Reference field → source record — and checks at each step whether that user actually has access. If they don't have visibility on the referenced record, the value is hidden. This all happens server-side, so the API and every client are protected automatically with no UI changes needed.

Pros

  • Protects API and client does not need to know about this

Cons

  • May be complicated to implement this multi-object traversal in AC
  • Does this actually solve the issue? The read-path is fixed
